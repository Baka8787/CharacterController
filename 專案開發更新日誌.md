
# 專案開發更新日誌 (Changelog & Learning Record)

---

## [v0.1] - 2026-06-28

### 地基建設與基礎資料流驗證

#### 1. 變更內容

* **核心資料結構建立**：實作了 `InputData` 類別、`IntentData` 結構體與 `PlayerRuntimeData`（黑板）。


* **輸入源解耦**：定義了 `IInputSource` 介面，並透過 Unity New Input System 實作了 `PlayerInputSource`。


* **管線驅動核心**：建立 `CharacterPipelineRunner`，在 `Update` 中依序執行「採樣 $\rightarrow$ 轉化意圖 $\rightarrow$ 處理參數」，並在 `LateUpdate` 清空意圖。


* **除錯可視化**：實作 `BlackboardDebugViewer` 透過 OnGUI 即時監控數據流。



#### 2. 設計理由與學習點

* **意圖與參數分離**：`IntentData` 處理單幀觸發事件（如跳躍），`PlayerRuntimeData` 的其餘欄位處理連續數值（如移動速度），這能讓狀態機與動畫層各自獲取乾淨的資料。


* **結構體整體覆寫（零 GC）**：`IntentData` 採用 `struct` 設計，在幀尾透過 `Reset()` 覆寫欄位，完全避免了每幀產生的記憶體垃圾（GC Alloc）。


* **單一實例複用**：`PlayerInputSource` 內部持久持有一個 `private readonly InputData`，每幀只更新其欄位而非 `new` 出新物件，同樣是為了效能優化。

## [v0.2] - 2026-06-29

### 命名規範落實與架構防禦加固

#### 1. 變更內容

* **程式碼命名規範嚴格化**：私有欄位全面重構為 `_camelCase`（例如 `_inputData`、`_runtimeData`），公開屬性與欄位維持 `PascalCase`。


* **唯讀引用防護（黑板封裝）**：擴充黑板欄位，新增 `CurrentWeapon` 與 `AimTarget`。將 `CurrentWeapon` 的 setter 權限設為 `internal`，嚴格限制外部模組的修改權限。


* **基礎 Dummy 類別補全**：新增 `ItemInstance` 空類別，確保專案在地基階段能直接通過編譯。


* **管線順序防禦（註解與結構鎖定）**：在 `CharacterPipelineRunner` 中寫入明確的順序警語，防止未來重構時將 `Intent.Reset()` 誤提早執行。


* **除錯監視器升級**：更新 `BlackboardDebugViewer`，將 v0.2 新增的引用區（武器、瞄準目標）納入 GUI 畫面的監視範圍。

#### 2. 設計理由與學習點

> ### 💡 關鍵架構思考：為什麼不現在就把 `InputData` 改成 `ref struct`？
> 
> 
> 雖然 v0.2 文檔中提到了 `ref struct` 的重構規劃，但我們選擇**暫不實作**。
> 
> 
> * **理由**：`ref struct` 雖然能徹底消除「鬼影資料（跨幀持有過期資料）」的風險，但它帶有極大的限制（不能當作 class 的欄位、會造成介面簽名的破壞性變更）。在資料流尚未完全穩定前，過早優化（Premature Optimization）會大幅降低架構調整的彈性。
> 
> 
> 
> 

* **防範 C# 結構體複製陷阱**：在 `PlayerRuntimeData` 中，`Intent` 欄位必須維持 **公開欄位 (Public Field)** 形式。如果將其改成公開屬性 (Property)，C# 的 Property 語意會觸發 `struct` 的值複製機制，導致你寫出 `data.Intent.JumpRequested = true` 時會直接編譯失敗（無法修改表達式產生的副本）。


* **維護資料寫入邊界**：遵循規格書「新增欄位必須明確誰寫誰讀」的規範，透過 `internal set` 鎖定寫入權，防止日後多處模組同時改寫同一數據而導致 Race Condition（競爭條件）或難以追蹤的 Bug。


* **守護順序脆弱點**：管線化架構（Pipeline）最怕執行順序錯亂。如果 `Intent.Reset()` 被移到狀態機讀取之前，玩家的輸入意圖就會直接被「吃掉」。這次更新特地在 `LateUpdate` 設立防禦註解，是為了給未來的自己（或團隊成員）打預防針。

## 3. 未來重構訊號（Refactoring Triggers）

當你在未來的第 3-4 週開發中看到以下現象，請立刻翻閱 `02-dev-spec_2.md` 的規劃草案進行重構：

1. **處理器肥大**：`CharacterPipelineRunner` 內的 `ProcessIntents` 或 `ProcessParameters` 超過 10 行時 $\rightarrow$ 立刻執行 **3.1 節**，將處理器抽離成 `IIntentProcessor` 介面群。


2. **跨幀持有風險**：發現有其他新寫的模組嘗試悄悄把 `InputData` 存成成員變數時 $\rightarrow$ 立刻執行 **1.3 節**，將 `InputData` 強制轉型為 `ref struct`，利用編譯器幫你擋下這種不安全的行為。
